<a name="introduction"/>

## Введение

Данный документ описывает план тестирования приложения **"ResourceHub"**.  
Документ предназначен для тестировщиков и разработчиков, участвующих в проверке корректности и стабильности работы системы.  
Цель тестирования — убедиться, что функционал приложения соответствует техническим требованиям и пользовательским ожиданиям.

<a name="items"/>

## Объект тестирования

В рамках тестирования будут проверены следующие функциональные модули системы:

* Регистрация и авторизация пользователей;
* Добавление, редактирование и удаление ресурсов;
* Просмотр списка ресурсов;
* Лайки/анлайки ресурсов;
* Управление пользователем (профиль);
* Навигация по страницам приложения;
* Обработка ошибок и уведомления.

<a name="quality"/>

## Атрибуты качества

1. **Функциональность**
    - Приложение выполняет все заявленные функции корректно.
    - Все API-запросы возвращают корректные ответы и коды состояния.

2. **Удобство использования**
    - Интерфейс понятен, доступен и адаптирован под разные размеры экранов.
    - Минимальное количество действий для выполнения основных операций.

3. **Производительность**
    - Время отклика API не превышает 3 секунд.
    - Интерфейс не зависает при загрузке данных.

4. **Безопасность**
    - Доступ к ресурсам возможен только после авторизации.
    - Токены и пароли защищены.

<a name="risk"/>

## Риски

Основные риски при тестировании:

* Некорректная работа авторизации (утечка токена, пропуск валидации);
* Потеря данных при удалении или редактировании ресурса;
* Ошибки при интеграции фронтенда с backend API;
* Некорректное отображение интерфейса на разных устройствах;
* Неправильная обработка ошибок сервера.

<a name="features"/>

## Аспекты тестирования

Проверке подлежат следующие ключевые сценарии:

### Регистрация и авторизация пользователей
* Успешная регистрация нового пользователя;
* Обработка пустых и некорректных данных при регистрации;
* Авторизация с корректными данными;
* Ошибка при неверном пароле;
* Поведение при истёкшем токене.

### Управление ресурсами
* Добавление нового ресурса;
* Проверка обязательных полей при добавлении;
* Редактирование ресурса и проверка сохранения изменений;
* Удаление ресурса;
* Отображение актуального списка после операций.

### Лайки и анлайки
* Возможность поставить лайк на ресурс;
* Повторное нажатие снимает лайк;
* Проверка отображения количества лайков;
* Поведение при попытке лайка без авторизации.

### Навигация и UI
* Проверка переходов между страницами ("Главная", "Мои ресурсы", "Профиль");
* Центрирование контента и адаптивность;
* Корректная работа бокового меню и шапки;
* Отображение уведомлений об ошибках и успешных действиях.

### Работа с API
* Проверка ответов от `/auth/register` и `/auth/login`;
* Проверка CRUD-запросов к `/resources`;
* Проверка обработки сетевых ошибок и статусов 400/401/404/500.

<a name="approach"/>

## Подходы к тестированию

Применяются следующие типы тестирования:

* **Модульное тестирование** — проверка отдельных сервисов и контроллеров backend'а;
* **Интеграционное тестирование** — взаимодействие фронта с backend API;
* **Системное тестирование** — комплексная проверка всей системы;
* **UI-тестирование** — проверка визуальных и интерактивных элементов интерфейса;
* **Приемочное тестирование** — проверка функционала с точки зрения конечного пользователя.

<a name="pass"/>

## Представление результатов

Результаты тестирования будут оформлены в таблице, содержащей:

* Наименование тестового сценария;
* Выполненные действия;
* Ожидаемый результат;
* Фактический результат;
* Итог теста (пройден / не пройден);
* Список найденных багов с приоритетами.

<a name="conclusion"/>

## Выводы

Данный план тестирования охватывает весь ключевой функционал проекта **ResourceHub**.  
Его выполнение позволит убедиться в корректной работе приложения, устойчивости к ошибкам и удобстве использования.  
Результаты тестирования помогут своевременно выявить и устранить дефекты перед релизом новой версии системы.
